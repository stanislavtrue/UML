<h2 align="center">Завдання №1</h2>

### Умова:
Побудуйте діаграму варіантів використання ***(Use Case)*** для простої системи:
«Система бронювання квитків у кінотеатрі».

**Вимоги:**
Мінімум 2 актори (наприклад, Клієнт, Адміністратор). Не менше 5 варіантів використання (бронювання, оплата, скасування, вхід у систему, перегляд сеансів). Використати зв’язки include та extend (наприклад, Оплата → include → Перевірка картки). 

**Очікуваний результат:** 
діаграма з чітко підписаними акторами і варіантами використання.

<h2 align="center">Завдання №2</h2>

### Умова:
Побудуйте діаграму класів ***(Class)*** для простої бібліотечної системи:
«Система керування бібліотекою» (читачі, книги, видача, адміністратор).

**Вимоги:** 
Не менше 4 класів (наприклад, Book, Reader, Loan, Librarian). Для кожного класу визначити 2–3 атрибути й 1–2 методи. 

**Показати зв’язки:**

``Reader — Loan (1..*)``, ``Book — Loan (1..*)``, ``Librarian — Book`` (керування каталогом).

**Додати тип зв’язку:** association, aggregation або composition.

**Очікуваний результат:** діаграма з атрибутами, методами й типами зв’язків.

<h2 align="center">Завдання №3</h2>

### Умова
Побудуйте діаграму послідовностей ***(Sequence)*** для сценарію:
«Користувач оформлює замовлення в онлайн-магазині».

**Кроки:** Користувач вибирає товар. Система створює замовлення. Сервіс оплати підтверджує транзакцію. Система надсилає email з підтвердженням.

**Об’єкти:**
``User``, ``WebApp``, ``OrderService``, ``PaymentService``, ``EmailService``.

**Очікуваний результат:** стрілки викликів між об’єктами, пунктирні зворотні відповіді.

<h2 align="center">Завдання №4</h2>

### Умова
Побудуйте діаграму діяльності ***(Activity)*** для сценарію:
«Реєстрація користувача у вебсистемі».

**Кроки:** Введення даних користувача. Перевірка валідності. Якщо дані некоректні — повідомлення про помилку. Якщо дані коректні — створення облікового запису. Надсилання листа з підтвердженням.

**Очікуваний результат:** чіткий потік дій із розгалуженням *(decision diamond)*.
